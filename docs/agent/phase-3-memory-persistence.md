# 第三阶段：会话、主题与压缩记忆数据底座

## 1. 阶段目标

本阶段在独立的 `ticket-agent-service` 中建立可审计、可重试的会话记忆底座，不调用票务业务接口，也不实现对话编排和 MCP 工具。

完成后的数据链路为：

1. 用户问题先写入原始消息表，并通过请求标识保证轮次幂等。
2. 主题判定只加载当前问题、最近用户问题和最近主题摘要卡片。
3. 选定主题后，再加载该主题的最新完整摘要和预算内最近完整消息。
4. 上下文只保存引用、消息范围和哈希快照，不复制消息正文。
5. 达到阈值后创建持久化摘要任务，由独立线程池异步处理。
6. 摘要成功后原子写入新版本并推进主题压缩边界；失败则延迟重试，原始消息始终保留。

## 2. 数据表

Flyway 脚本 `V1__create_agent_memory_tables.sql` 创建以下 9 张表：

| 表名 | 职责 |
| --- | --- |
| `t_agent_conversation` | 会话所有权、状态、活动主题和消息序号分配 |
| `t_agent_topic` | 会话内主题、路由短摘要、结构化状态和摘要边界 |
| `t_agent_turn` | 一轮用户问题到最终助手回答的状态与幂等信息 |
| `t_agent_message` | 不可变的用户、助手和工具原始消息 |
| `t_agent_memory_summary` | 可回放的主题摘要版本 |
| `t_agent_summary_task` | 摘要任务状态、租约、重试和失败审计 |
| `t_agent_context_snapshot` | 实际装配上下文的引用、范围、Token 估算和哈希 |
| `t_agent_context_route_log` | 主题候选、选择结果、置信度和模型调用引用 |
| `t_agent_model_call` | 模型平台、候选、耗时、Token 和失败分类审计 |

表之间暂不建立数据库外键，服务层负责会话所有权和聚合一致性校验，避免后续归档及分库时受到级联约束。

## 3. 主题上下文策略

### 3.1 判定主题

主题判定输入严格限制为：

- 当前用户问题；
- 最近若干条历史用户问题；
- 最近若干个活动主题的短摘要卡片。

此步骤不加载历史助手回答和完整主题消息，降低无关上下文干扰及模型成本。

### 3.2 加载完整上下文

主题选定后加载：

- 该主题最新活动摘要；
- 摘要边界之后的最近完整消息；
- 主题结构化业务状态。

消息同时受数量上限和 Token 预算约束。系统为每个请求保存首次上下文快照和首次路由日志，同一请求重试不会重复写入审计记录。

## 4. 摘要任务状态机

状态流转如下：

```text
PENDING -> RUNNING -> SUCCEEDED
                   -> RETRY_WAIT -> RUNNING
                   -> FAILED
```

- 领取任务时增加尝试次数并建立有限租约。
- 工作节点异常退出后，租约到期的 `RUNNING` 任务可以被其他节点接管。
- 失败重试采用随尝试次数递增的等待时间。
- 摘要提交时校验期望版本，只允许连续推进主题摘要版本和消息边界。
- 摘要失败不会删除原始消息，也不会阻塞当前用户回答。

`SummaryTaskProcessor` 是下一阶段接入低成本摘要模型的端口。本阶段没有实现该端口时，异步分发器不会领取任务，避免任务在无处理器环境中产生无意义失败。

## 5. 运行配置

生产环境默认连接本机 MySQL 的 `12306_agent` 数据库，建议显式配置：

```text
AGENT_DATASOURCE_URL=jdbc:mysql://127.0.0.1:3306/12306_agent?useUnicode=true&characterEncoding=UTF-8&serverTimezone=Asia/Shanghai&useSSL=false&allowPublicKeyRetrieval=true
AGENT_DATASOURCE_USERNAME=root
AGENT_DATASOURCE_PASSWORD=数据库密码
```

数据库本身需要预先创建，应用启动时 Flyway 自动创建和校验业务表。测试环境使用 H2 的 MySQL 兼容模式，不访问生产数据库。

可通过 `index12306.agent.memory` 调整主题候选数、最近消息数、上下文 Token 预算、摘要触发阈值、重试次数、租约和专用线程池容量。

## 6. 阶段边界

本阶段尚未包含：

- 主题判定和摘要生成的真实模型调用；
- Spring AI 对话编排；
- MCP 票务查询、购票、退票工具；
- 面向客户端的聊天接口和流式输出；
- 已有多模型路由审计记录与请求上下文的最终串联。

这些内容应在后续阶段基于本阶段服务接口继续实现，而不是绕过持久化状态直接操作数据库。
